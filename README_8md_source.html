1 # GlobalCovfefe -- Emulation of a GlobalCache networked IR device
2 
3 The present program/library emulates a TCP commands of a
4 [GlobalCach√©](https://www.globalcache.com)
5 networked IR sender, like the [Flex](https://www.globalcache.com/products/flex/).
6 The emulation is not complete. The device emulated has one IR sender,
7 and optionally a learner. Sensor inputs and relay closures are planned,
8 but not yet implemented.
9 
10 Hardware requirements are a "sufficiently large" Arduino with
11 a supported Ethernet shield.
12 The program has been developed on an Arduino Mega 2560 with an Ethernet shield
13 from the first generation (with W5100 chip). It is believed, that, possibly after
14 some stripping, it should run on an Uno-class board. However, this has
15 not yet been verified.
16 
17 The program/library is built on top of my IR library
18 [Infrared4Arduino](https://github.com/bengtmartensson/Infrared4Arduino).
19 Optionally, the [ABeacon](https://github.com/bengtmartensson/ABeacon)
20 library is used to advertise the device on the local network.
21 Other that that, only official Arduino libraries are used.
22 
23 The TCP API is described
24 [here](https://www.globalcache.com/files/releases/flex-16/API-Flex_TCP_1.6.pdf).
25 
26 The program/library was written from scratch by myself. It is licensed under the

29 ## Software architecture
30 There is a class `GlobalCovfefe` and a derived class `GlobalCovfefeLearner`
31 contained in the `src` directory. These classes operates
32 on a [Stream](https://www.arduino.cc/reference/en/language/functions/communication/stream/).
33 This is typically, but not necessarily,
34 an [`EthernetClient`](https://www.arduino.cc/en/Reference/ClientConstructor) instance.
35 
36 ### Example `GlobalCovfefe`
37 The "example" `GlobalCovfefe` contains the deployment program. It controls the
38 Ethernet connections, and delivers it, as a `Stream` to an instance of the class `GlobalCovfefe[Learner]`.
39 
40 ### Example SerialCovfefe
41 The example `SerialCovfefe`, however, uses only `Serial`, and can be used for
42 testing or debugging the library.

44 ### Single-threaded operation
45 Most important difference from the GlobalCache devices
46 is that the current program is single-threaded.
47 That is, it accepts one command, executes it, and only when it is finished
48 (or times out), the program listens for new commands.
49 Thereby, commands like `stopir` are not meaningful, and not implemented.
50 
51 #### Only one concurrent TCP session
52 For this reason, only one connection at a time is possible.
53 If one connection is active, the device will not respond to further connection
54 attempts until the first connection is terminated. (This makes a LED indicating
55 open connection () quite useful.)
56 
57 ## Commands implemented
58 * `getversion`: Returns the version of the current program, identical to the version in the library manager.
59 * `getdevices`: Returns the configured devices in a machine parseable way.
60 * `blink`: Blinks the configured LEDs.
61 * `sendir`: sends a (raw) IR command. Compressed commands are not implemented ([but planned](https://github.com/bengtmartensson/GlobalCovfefe/issues/7)).
62 * `get_IRL`: reads an IR signal and returns it (Only if a learner has been configured.)
63 
64 Expected responses are produced.
65 Error handling and sensibility checking of the provided arguments
66 are essentially non-existent.
67 
68 ## Differences to the GlobalCach√© Flex
69 * Only a subset of the TCP commands are implemented.
70 * The HTTP API, including file storage, is not supported.
71 * Only one sender is supported.
72 * Configuration is during compilation only; the GlobalCach√© products are higly
73 configurable, for example with an embedded HTTP server.
74 * Only one session, single-threaded. (Thus no commands like `stopir` would be sensible.)

81 ## Configuration
82 The GlobalCache hardware is extensively configurable, both using the TCP commands,
83 as well as using the built-in HTTP-server. This program is instead configured
84 during compilation, using CPP defines in the file `config.h`. These are described in that file.
85 
86 For the configuration of the IrSender and, optionally, the learner, see the
87 documenation of Infrared4Arduino.
88 
89 ### Signaling LEDs
90 The library supports up to three LEDs, indicating the present operating mode.
91 These are defined in `config.h` by assigning pin numbers (denoting pins that make
92 "something" light up when make high) to CPP symobls. This is highly recommended
93 for problem solving and debugging.

115 ## Porting
116 GlobalCovfefe as such is a portable program, not a program that needs porting.
117 There are no other "dependencies" than a certain resource consumption.
118 Resources are allocated only indirectly. through the used libraries.
119 However, Infrared4Arduino (which is the low-level library that this one builds upon)
120 only runs
121 on some hardware, and does allocate some resources like timers and interrupts.
122 Porting it it to more hardware platforms is desirable and planned.
123 Help is solicited.

130 ## Use with IrScrutinizer
131 Unfortunately, the restriction of only one session can cause somewhat unexpected
132 behavior in [IrScrutinizer](https://github.com/bengtmartensson/harctoolbox).
133 If opened for sending (using the "Sending hw" pane), the connection has to be
134 closed before opening for capturing (the "Capturing hw" pane) --
135 and similarly for going from capturing to transmitting.
136 Unfortunately, in the present IrScrutinizer there is no possibility to force
137 a GlobalCache connection to close, except for opening a connection with another
138 GlobalCache device, which is of course not possible if only one is present...
139 ([IrScrutinizer issue](https://github.com/bengtmartensson/harctoolboxbundle/issues/246))
140 Workaround: reset the Arduino to close the connection.
